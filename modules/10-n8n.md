## Week 25 — n8n Fundamentals + Claude Integration

### What n8n is, in one sentence
**n8n is a self-hosted workflow automation tool** (think Zapier, but open source, runs on your infra, code-extensible). Nodes represent actions; edges represent data flow. You chain them visually.

### Why self-hosted matters here
You're building AI automations that touch code, test runners, internal APIs, and credentials. You don't want that data flowing through a third-party SaaS. n8n Cloud exists, but for this module we'll self-host.

### Spin it up (30 min)

```bash
mkdir n8n-local && cd n8n-local
cat > docker-compose.yml << 'EOF'
version: '3.8'
services:
  n8n:
    image: docker.n8n.io/n8nio/n8n
    restart: always
    ports:
      - "5678:5678"
    environment:
      - N8N_BASIC_AUTH_ACTIVE=true
      - N8N_BASIC_AUTH_USER=admin
      - N8N_BASIC_AUTH_PASSWORD=changeme
      - N8N_ENCRYPTION_KEY=your-random-32-char-key-here
    volumes:
      - n8n_data:/home/node/.n8n

volumes:
  n8n_data:
EOF

docker compose up -d
# Open http://localhost:5678
```

### Your first Claude workflow (2 hours)

**Build:** A Slack → Claude → Slack bot that:
1. Triggers on a Slack message in `#qa-alerts`
2. Sends the message to Claude asking "classify this: bug / flaky test / environment issue / other"
3. Posts the classification + 1-line explanation back to the thread

**n8n nodes to use:**
- `Slack Trigger` → `HTTP Request` (Anthropic API) → `Code` (parse response) → `Slack`

**The HTTP Request node config:**
```json
{
  "method": "POST",
  "url": "https://api.anthropic.com/v1/messages",
  "headers": {
    "x-api-key": "={{ $env.ANTHROPIC_API_KEY }}",
    "anthropic-version": "2023-06-01",
    "content-type": "application/json"
  },
  "body": {
    "model": "claude-haiku-4-5-20251001",
    "max_tokens": 256,
    "messages": [
      {
        "role": "user",
        "content": "={{ $json.text }}"
      }
    ],
    "system": "Classify the QA alert as: bug | flaky_test | environment | other. Reply with JSON: {\"category\": \"...\", \"reason\": \"one sentence\"}."
  }
}
```

Use `claude-haiku-4-5-20251001` here — classification is simple enough that Haiku is ideal. Save Sonnet for tasks that actually need it.

### Key concepts to internalize

**n8n expressions** — `={{ }}` accesses data from previous nodes:
```
={{ $json.text }}              // current node's JSON
={{ $node["Claude"].json }}   // specific node by name
={{ $env.ANTHROPIC_API_KEY }} // environment variable
={{ $now.toISO() }}           // built-in helpers
```

**Error workflows** — wire every automation to an error handler:
```
[Main workflow] --error--> [Error Handler workflow]
                             → Slack alert: "Workflow X failed: {{ $json.error }}"
                             → Write to error log
```

**Retries** — configure on any node: max retries, wait between, continue on fail.

### 🛡️ Security callout
- Store credentials in n8n's built-in credential vault, **never** in node params
- n8n runs with your permissions — treat it like an MCP server: scope carefully
- If n8n is network-accessible, use basic auth + reverse proxy with TLS (e.g., Caddy)
- Never expose n8n webhooks without authentication

### Project A — Test failure triage pipeline (2 hours)

When CI fails, the QA team spends 30 minutes figuring out if it's a real bug or a flaky test. Automate the triage.

**Trigger:** GitHub webhook on `workflow_run` failed event

**Steps:**
1. Fetch the failed test names + error messages via GitHub API
2. Call Claude (Sonnet 4.6) with the failures + recent git diff
3. Claude classifies each failure: `real_bug | flaky | environment | needs_investigation`
4. Create a Jira ticket for `real_bug` failures (Jira node)
5. Post a triage summary to Slack with per-test verdicts
6. For `flaky` failures, add to a Google Sheet tracking flakiness rate

**Why this is valuable:** You just turned a 30-minute morning ritual into a 2-minute Slack message that appears automatically.

### Project B — n8n calling your MCP server (2 hours)

Your Week 14 MCP server exposed Playwright tools to Claude Desktop. Now expose them to n8n so non-technical stakeholders can trigger test runs.

**Step 1:** Wrap your MCP tools in a simple HTTP API:
```ts
// tools-api.ts — thin Express wrapper around your MCP logic
import express from "express";
import { runPlaywrightTest } from "./your-mcp-tools";

const app = express();
app.use(express.json());

// Require a shared secret header
app.use((req, res, next) => {
  if (req.headers["x-tools-secret"] !== process.env.TOOLS_SECRET) {
    return res.status(401).json({ error: "unauthorized" });
  }
  next();
});

app.post("/run-test", async (req, res) => {
  const { test_file } = req.body;
  const result = await runPlaywrightTest(test_file);
  res.json(result);
});

app.listen(4321);
```

**Step 2:** In n8n, create a "Run Test" workflow:
- Trigger: Webhook (so Jira/Slack/anyone can call it)
- Call: `HTTP Request → http://your-tools-api:4321/run-test`
- On pass: post green Slack message
- On fail: create Jira ticket with failure output

Now a QA manager can trigger a specific test suite from a Slack slash command — without a terminal.

### Project C — Nightly AI quality digest (1 hour)

**Schedule:** Every day at 8 AM

**Steps:**
1. Query your Langfuse dashboard API → last 24h stats
2. Query your pgvector eval results DB → latest pass rates
3. Query GitHub → open PRs with failing evals
4. Send all context to Claude Sonnet 4.6: "Write a 3-bullet daily digest"
5. Post digest to Slack `#ai-quality`

This is the kind of automation that makes your team say "where did this come from?" — and then you tell them it took you an hour.

### The boundary: n8n vs code agents

| Use n8n when... | Use a code agent when... |
|---|---|
| Connecting to SaaS tools (Jira, Slack, Sheets) | LLM needs to iterate / loop dynamically |
| Scheduling (cron, webhook trigger) | You need complex conditional logic |
| Non-engineers need to edit the flow | Evals gate deployment |
| Data pipelines with fixed steps | Multi-step tool use / ReAct patterns |
| Prototyping a new integration fast | The logic has evolving requirements |

**Rule of thumb:** if a business analyst could draw the flowchart and it has fewer than 15 steps, n8n wins. If the LLM needs to decide what to do next, write a code agent.
